# Emplay Reference Document:  AI Prompt Library Application

## Objective

The goal of this project is to build a full-stack web application that helps users store and manage AI image generation prompts.
In simple terms, this app acts like a small library where users can:

- view saved prompts
- add new prompts
- open one prompt to see its full details
- track how many times each prompt has been viewed

This project solves a common problem: keeping AI prompts organized instead of storing them in random notes, chats, or documents. By the end, you should have a working application with:

- an Angular frontend for the user interface
- a Django backend for handling data and APIs
- PostgreSQL for storing prompts permanently
- Redis for storing the view count efficiently
- Docker Compose to run everything together with one command

The final outcome should be a beginner-friendly but properly structured full-stack application that follows the assignment requirements.

You are expected to use the following stack:

- **Frontend:** Angular 14 or higher

## Step-by-Step Implementation Guide

Your application needs:

### 3. Create the backend project

Start by creating a Django project and an app.

Typical steps:

- create a Django project, for example backend/
- create an app, for example prompts/
- configure PostgreSQL in settings
- add the prompts app to INSTALLED_APPS

### 4. Design the Prompt Model

The assignment requires a Prompt model with:

- id
- title
- content
- complexity
- created_at

The model defines how your prompt data is stored in PostgreSQL.
Things to keep in mind:

- title should be a short text field
- content should be a longer text field
- complexity should be an integer
- created_at should be automatically filled when the prompt is created
- id can be either integer or UUID

**Example model snippet**

```
from django.db import models

class Prompt(models.Model):
    title = models.CharField(max_length=255)
    content = models.TextField()
    complexity = models.IntegerField()
    created_at = models.DateTimeField(auto_now_add=True)
```

**What to expect after this step:**

- you run migrations
- PostgreSQL gets a table for prompts
- you can add test data from Django admin or shell if needed

### 5. Add validation rules in the backend

Although the frontend will validate input, backend validation is still important.
 Frontend validation improves user experience, but backend validation protects your data. Someone could still send invalid API data directly.
Examples of rules:

- title should not be too short
- content should not be too short
- complexity must be between 1 and 10

You can validate in forms, model clean methods, or inside the view logic. 

For a beginner-friendly approach, validating inside request handling is acceptable if done carefully.
**Expected result:** the backend rejects bad input with helpful error messages.

### 6. Build the API endpoints in Django

The assignment asks for REST-style endpoints, but you should build them using plain Django views, not DRF.
You need:

- **GET /prompts/**
Returns all prompts. Used by the Angular list page.
- **POST /prompts/**
Creates a new prompt. Used by the Angular form page.
- **GET /prompts//**
Returns one prompt and increments its Redis view count. Used by the Angular detail page.

A simple way  is to use function-based views or class-based views with JsonResponse.

**Example endpoint snippet**

```
from django.http import JsonResponse
from .models import Prompt

def prompt_list(request):
    if request.method == "GET":
        prompts = Prompt.objects.all().values("id", "title", "complexity", "created_at")
        return JsonResponse(list(prompts), safe=False)
```

**Expected result:**

- visiting the endpoint returns JSON
- Angular will later be able to consume it

### 7. Connect Redis for the view counter

This is one of the most important parts of the assignment.
The rule is:

- every time a prompt is retrieved via GET /prompts/:id/
- increment the counter in Redis
- return view_count in the response
- Redis is the source of truth for view counts

A common pattern is:

- use a key like prompt::views
- increment it with Redis INCR
- read back the value
- include it in the JSON response

**Example snippet**

```
import redis

r = redis.Redis(host="redis", port=6379, db=0)

def get_prompt_views(prompt_id):
    key = f"prompt:{prompt_id}:views"
    return r.get(key)
```

And during detail retrieval:

```
key = f"prompt:{prompt.id}:views"
view_count = r.incr(key)
```

**What to expect:**

- every time the detail endpoint is called, the number goes up
- PostgreSQL remains unchanged
- the frontend sees live view count data

### 8. Test backend endpoints before building the frontend

Before working on Angular, make sure the backend works correctly.
If APIs are broken, frontend debugging becomes confusing.
Test all three endpoints using:

- browser for GET requests
- Postman, Insomnia, or curl for POST
- repeated calls to detail endpoint to confirm Redis counter increments

Check:

- list endpoint returns all prompts
- create endpoint stores a new prompt
- detail endpoint returns prompt data plus updated view_count

**Expected result:** backend is stable enough for frontend integration.

### 9. Create the Angular frontend

Now build the user interface.
You will typically create:

- an Angular project inside frontend/
- routing for navigation
- components for: prompt list, prompt detail, add prompt form
- a service for API calls

**Expected result:** Angular app starts successfully and can be opened in the browser.

### 10. Set up Angular routing

The frontend should support navigation between pages.
A simple route structure could be:

- /prompts → list view
- /prompts/:id → detail view
- /add-prompt → create form

Why this is needed: routing makes the app feel like a real frontend application with separate pages.
**Expected result:** clicking links changes views without reloading the whole app.

### 11. Create an API service in Angular

Instead of calling APIs directly inside every component, create one shared service.
Why this is needed:

- keeps components cleaner
- avoids repeated code
- makes API logic easier to update later

Typical responsibilities of the service:

- fetch all prompts
- fetch one prompt by ID
- create a new prompt

**Example service snippet**

```
getPrompts() {
  return this.http.get<any[]>('/api/prompts/');
}

getPrompt(id: string) {
  return this.http.get<any>(`/api/prompts/${id}/`);
}

createPrompt(data: any) {
  return this.http.post('/api/prompts/', data);
}
```

**Expected result:** components can call one service instead of manually handling URLs everywhere.

### 12. Build the Prompt List View

This page should fetch all prompts and display: title and complexity.
A good beginner-friendly interface might show:

- prompt title as main text
- complexity as a colored badge
- a button or link to open the detail page

You can visually indicate complexity using CSS classes such as:

- low complexity → green
- medium complexity → orange
- high complexity → red

**Expected result:** user can open the app and immediately see all saved prompts.

### 13. Build the Prompt Detail View

This page should:

- read the prompt ID from the route
- call the detail API
- display full prompt content
- display the live view_count

This is where the Redis integration becomes visible to the user.
Each time this page loads, the backend endpoint is called, so the view count increases automatically.
**Expected result:** opening the same prompt multiple times shows the count increasing.

### 14. Build the Add Prompt Form using Reactive Forms

The assignment specifically asks for a reactive form.
Reactive forms are the recommended Angular approach for structured validation and form control.
Your form should include: title, content, and complexity.

Validation rules:

- title minimum 3 characters
- content minimum 20 characters
- complexity between 1 and 10

**Example form setup snippet**

```
this.promptForm = this.fb.group({
  title: ['', [Validators.required, Validators.minLength(3)]],
  content: ['', [Validators.required, Validators.minLength(20)]],
  complexity: [1, [Validators.required, Validators.min(1), Validators.max(10)]]
});
```

**Expected result:**

- invalid form shows error messages
- valid form submits successfully
- new prompt appears in the list

### 15. Display clear validation messages

Do not just block submission silently.
Examples:

- “Title must be at least 3 characters.”
- “Content must be at least 20 characters.”
- “Complexity must be between 1 and 10.”

**Expected result:** users understand how to fix their input.

A proxy is often easier in development because it avoids CORS issues.
**Expected result:**

- list page fetches backend data

### 19. Create the docker-compose.yml file

This file is the glue that connects everything.
It should define services for: frontend, backend, postgres, redis.
Rthe assignment requires the full stack to run with one command:
`docker-compose up --build`

Important responsibilities of Compose:

- create shared network
- provide environment variables
- expose ports
- ensure containers can talk to each other

**Expected result:** one command starts the complete system.

**Example compose idea**

```
services:
  db:
    image: postgres:14
  redis:
    image: redis:7
  backend:
    build: ./backend
  frontend:
    build: ./frontend
```

This is only a simplified example, not a full config.

### 20. Handle database migrations and startup flow

A common beginner issue is forgetting migrations inside Docker.
Why this is needed: the app cannot work until PostgreSQL has the correct tables.
Your backend startup process should make sure:

- database is reachable
- migrations are applied

**Expected result:** when Docker Compose runs, the database is ready and Django works without manual repair steps.
